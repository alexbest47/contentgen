# Список изменений для совместимости экспорта лендингов с WordPress

## Контекст

Программист прислал обратную связь по формату экспорта ZIP из конструктора лендингов. Цель — чтобы выгруженный файл сразу работал как шаблон WordPress-страницы на talentsy.ru без ручных правок. Ниже — полный список расхождений между текущим экспортом и требованиями WP-стороны, сгруппированный по областям.

---

## 1. Формат выходного файла: PHP вместо HTML

**Сейчас:** экспорт генерирует `index.html` — статичную HTML-страницу.

**Нужно:** экспорт должен генерировать `.php`-файл (например, `na-{slug}.php`), который является WordPress Page Template.

**Что менять:**
- В Edge Function `export-landing` изменить расширение выходного файла с `.html` на `.php`
- Имя файла формировать по паттерну `na-{slug}.php`, где slug берётся из названия лендинга (транслитерация + kebab-case)

---

## 2. PHP-шапка шаблона (Template Name)

**Сейчас:** файл начинается с `<!doctype html>`.

**Нужно:** файл должен начинаться с PHP-комментария, объявляющего шаблон WordPress:

```php
<?php
/*
Template name: __NewAge: {Название лендинга}
*/
```

**Что менять:**
- Добавить генерацию PHP-шапки в начало файла
- При создании нового лендинга — генерировать название автоматически
- При обновлении существующего — **не менять** название шаблона (хранить его в БД)
- Добавить поле `wp_template_name` в таблицу `landings` для хранения зафиксированного имени шаблона

---

## 3. Header: заменить HTML на PHP-вызов `get_header('light')`

**Сейчас:** в экспорте весь HTML header (doctype, head, meta-теги, CSS-линки, body-открытие, шапка сайта) генерируется целиком как статичный HTML.

**Нужно:** вместо всего этого блока в PHP-файле должна быть одна строка:

```php
get_header('light');
```

**Для превью в конструкторе:** содержимое `header-light.txt` подставляется как HTML, чтобы пользователь видел реальную шапку. Но в экспорте — только PHP-вызов.

**Что менять:**
- Сохранить содержимое `header-light.txt` в БД или как статичный ресурс для preview
- В `useLandingPreviewHtml.ts` (buildPreviewHtml) — подставлять HTML из header-light для превью
- В `export-landing` Edge Function — вставлять `get_header('light');` вместо всего HTML-header
- Убрать хардкод CSS/JS-ссылок из экспорта (они уже подключаются через WP header)

---

## 4. Footer: заменить HTML на PHP-вызов `get_footer('light')`

**Сейчас:** footer генерируется как статичный HTML (или вообще отсутствует).

**Нужно:** в конце PHP-файла должно быть:

```php
get_footer('light');
```

**Для превью:** аналогично header — подставлять HTML из `footer-light.txt`.

**Что менять:**
- Сохранить содержимое `footer-light.txt` для preview
- В экспорте — вставлять `get_footer('light');` вместо HTML footer
- В превью — показывать реальный footer-light HTML

---

## 5. Хлебные крошки (Breadcrumbs)

**Сейчас:** хлебных крошек нет.

**Нужно:** сразу после `get_header('light')` вставлять блок breadcrumbs:

```php
get_template_part('inc/landing-breadcrumbs', null, ['psychology', 'Курсы психологии']);
```

Параметры (slug категории и название) должны зависеть от выбранной программы.

**Что менять:**
- Добавить в модель лендинга (или программы) поля для breadcrumbs: `breadcrumb_slug` и `breadcrumb_title`
- Или формировать автоматически из привязанной программы
- В экспорте — вставлять PHP-вызов `get_template_part`
- В превью — подставлять HTML из `landing-breadcrumbs.txt` с подменой параметров

---

## 6. Формы: PHP-обработка вместо статичного HTML

**Сейчас:** формы в блоках — статичный HTML без серверной обработки.

**Нужно:** формы должны использовать PHP-переменные для обработки ($formParmas), как в эталоне:

```php
<form <?= $formParmas['tag']; ?> class="form-get form-get_new form-get_inherit <?= $formParmas['class']; ?>">
  <input type="text" name="<?= $formParmas['names']['name']; ?>" ... />
  <input type="number" name="<?= $formParmas['names']['phone']; ?>" ... />
  <input type="email" name="<?= $formParmas['names']['email']; ?>" ... />
  <? get_template_part( 'inc/components/form/agreed' ); ?>
  <?= $formParmas['hiddens']; ?>
</form>
```

**Что менять:**
- Блоки с формами в `landing_block_definitions` должны иметь **две версии HTML**: одну для preview (чистый HTML с agreed-блоком из `agreed.txt`), другую для экспорта (PHP с переменными `$formParmas`)
- Или: хранить единый шаблон с плейсхолдерами типа `{{FORM_TAG}}`, `{{FORM_NAME_FIELD}}`, `{{FORM_PHONE_FIELD}}`, `{{FORM_EMAIL_FIELD}}`, `{{FORM_AGREED}}`, `{{FORM_HIDDENS}}`, которые резолвятся по-разному для preview и export
- Код обработки форм ($isGetCourse и $formParmas) **запрещено менять** — он копируется из эталона as-is в начало шаблона после header

---

## 7. Блок обработки форм в начале шаблона

**Сейчас:** отсутствует.

**Нужно:** после `get_header('light')` и breadcrumbs, перед контентом, должен быть PHP-блок с переменными формы:

```php
$isGetCourse = get_field('is_gc');
if( $isGetCourse ):
    $formParmas = [ ... GetCourse конфигурация ... ];
else:
    $formParmas = [ ... стандартная конфигурация ... ];
endif;
```

**Что менять:**
- Этот PHP-блок неизменяемый — копируется из эталона в каждый экспорт as-is
- Хранить его как константу/шаблон в Edge Function

---

## 8. Пути к статическим ресурсам

**Сейчас:** в экспорте пути к CSS/JS относительные (`css/fix.css`, `js/landing.min.js`), предполагающие наличие рядом в ZIP.

**Нужно:**
- Пути к **существующим ресурсам темы** — относительные от корня домена: `/wp-content/themes/talentsy/...`
- Пути к **новым изображениям** (загруженным/сгенерированным) — полные URL с внешнего хранилища (Supabase Storage)
- В превью — подставлять домен `https://talentsy.ru` перед относительными путями

**Что менять:**
- В HTML-шаблонах блоков (`landing_block_definitions.html_template`) пути к штатным ресурсам темы должны быть от корня: `/wp-content/themes/talentsy/new-age/dist/images/...`
- При экспорте: относительные пути оставлять как есть, для новых изображений из Supabase Storage — полный URL
- В превью (`useLandingPreviewHtml.ts`): добавлять `https://talentsy.ru` к относительным путям
- Убрать из ZIP папки css/, js/, fonts/ — они не нужны, всё подключается через WP header/footer

---

## 9. Inline-стили вместо отдельного CSS

**Сейчас:** кастомные стили блоков (background_color, card_color) инжектируются как `<style>` в head или перед блоком.

**Нужно:** все кастомные стили добавляются **inline в конце шаблона, до `get_footer('light')`**. Классы для кастомных элементов должны быть **неконфликтными** (использовать уникальные суффиксы, как в эталоне: `_hg8ap7v1d`).

**Что менять:**
- Собирать все кастомные `<style>` блоки и размещать единым блоком перед `get_footer('light')`
- CSS-классы, создаваемые для кастомизации, должны содержать уникальный суффикс лендинга (можно использовать короткий хеш от ID лендинга)
- В превью — поведение аналогичное

---

## 10. Убрать ACF-переменные

**Сейчас:** не актуально (в текущем конструкторе их нет).

**Нужно:** убедиться, что в экспорте **нет** обращений к ACF-полям (`get_field()`, `get_fields()`), кроме формы ($isGetCourse, $formParmas). Программист пишет: «Планируется отказаться от получения данных из панели управления, всё хардкодом из редактора».

**Что менять:**
- Весь контент (преподаватели, FAQ, цены, программы) — хардкодом из конструктора
- Данные преподавателей, которые в эталоне грузятся через `$teachers` из ACF, в нашем экспорте должны быть вставлены как статичный HTML
- Аналогично $programs, $faq, $coursePrices — всё заменяется контентом из конструктора

---

## 11. Структура экспортируемого ZIP

**Сейчас:** ZIP содержит `index.html` + README.md, ресурсы не включены.

**Нужно:**
- Основной файл: `na-{slug}.php` — WordPress Page Template
- Папка `img/` — только новые изображения (загруженные/сгенерированные в конструкторе), которых нет на сервере talentsy.ru
- README.md с инструкцией: куда положить PHP-файл и изображения

**Что менять:**
- Переименовать выходной файл
- Убрать из ZIP ненужные папки (css/, js/, fonts/)
- Добавить в ZIP только новые изображения, которых нет в теме WP
- Обновить README с инструкцией по деплою

---

## 12. Двойной режим рендеринга блоков (Preview vs Export)

**Ключевое архитектурное изменение.** Сейчас есть единый HTML-шаблон блока. Нужна система, которая из одного шаблона генерирует два варианта:

**Preview (для конструктора):**
- Полный HTML с реальным контентом
- Header/Footer из txt-файлов (header-light, footer-light)
- Формы с HTML из agreed.txt
- Пути к изображениям — абсолютные (https://talentsy.ru/...)

**Export (для WordPress):**
- PHP-файл с WP-вызовами (get_header, get_footer, get_template_part)
- Формы с PHP-переменными ($formParmas)
- Пути к штатным ресурсам — от корня домена
- Inline-стили собраны в конце файла

**Что менять:**
- Добавить в `landing_block_definitions` поле `php_template` или систему плейсхолдеров для WP-специфичных частей
- Или: реализовать трансформацию HTML→PHP при экспорте (замена маркеров форм на PHP-код)
- Обновить Edge Function `export-landing` для генерации PHP
