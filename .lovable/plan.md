
Исправлю это как баг логики синхронизации в `EmailBuilder`, а не как баг генерации.

### Что реально ломается
Сейчас в `src/pages/EmailBuilder.tsx` есть две опасные вещи:

1. Инициализация данных из базы помечает письмо как `unsaved`, хотя пользователь ничего не менял.
2. Эффект `useEffect(() => () => { save(); }, [save])` вызывает `save()` не только при выходе со страницы, но и при каждом изменении зависимостей `save`.

Из-за этого происходит такой сценарий:
- письмо открывается и контент подгружается;
- параллельно один из cleanup-вызовов сохраняет старое/пустое состояние;
- в базе `generated_html` снова становится пустым;
- пока страница открыта, контент ещё виден из локального state;
- при повторном входе письмо уже пустое, потому что пустое состояние успело сохраниться в базу.

Это совпадает с симптомом: “увидел контент → быстро вышел → зашёл снова → контент пропал”.

### План исправления

#### 1. Разделить “загрузку из базы” и “пользовательские изменения”
В `src/pages/EmailBuilder.tsx`:
- добавлю флаг гидратации/синхронизации;
- значения из базы будут записываться в state без пометки `dirty`;
- `dirtyRef.current = true` будет ставиться только после реального пользовательского изменения.

Это уберёт ложное автосохранение сразу после открытия письма.

#### 2. Исправить unmount-save, чтобы он срабатывал только при выходе со страницы
Заменю текущий cleanup-эффект на безопасный паттерн:
- сохраню актуальную функцию `save` в `ref`;
- сделаю отдельный `useEffect(..., [])`, который вызывает сохранение только на реальном unmount;
- уберу повторные cleanup-save на каждый ререндер.

Это главное исправление, чтобы локальное пустое состояние больше не перетирало результат генерации.

#### 3. Сделать безопасное принятие данных из базы после генерации
Сейчас локальный state инициализируется только один раз (`initialLoadRef`), поэтому позднее обновление записи из базы может быть проигнорировано.

Исправлю так:
- если письмо пришло из базы с новым `updated_at` и локально нет несохранённых изменений, state будет обновляться;
- если в базе уже есть `generated_html`, а локально оно пустое, приоритет будет у данных из базы.

Это стабилизирует повторный вход в письмо и открытие после фоновой генерации.

#### 4. Добавить защиту от перезаписи готового письма пустым HTML
В `save()` добавлю safeguard:
- если письмо уже в состоянии `ready`, а локальный `generatedHtml` пустой из-за несинхронизированного state, не отправлять пустой `generated_html` в update;
- аналогично не затирать `image_placeholders` пустым массивом без явного пользовательского действия.

Это дополнительная страховка даже если где-то останется гонка.

### Что изменится в коде
Основной файл:

Вероятно без изменений базы и без изменений backend-функций.

### Проверка после исправления
Проверю такой сценарий:
1. сгенерировать письмо;
2. открыть его из очереди;
3. сразу выйти;
4. снова открыть письмо;
5. убедиться, что `generated_html`, тема, прехедер и изображения не исчезают;
6. отдельно проверить ручное редактирование письма и автосохранение через 30 секунд.

### Технически
Ключевая проблема не в самой генерации, а в комбинации:
- `initialLoadRef.current = true` + общий dirty-effect,
- cleanup с зависимостью `[save]`,
- сохранение всего письма целиком даже при частично устаревшем локальном state.

Исправление будет в сторону “hydrate safely, save only on user changes, unmount-save only once”.
