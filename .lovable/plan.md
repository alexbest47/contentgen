

## Диагноз: Edge Function timeout при генерации изображений

### Что происходит

Логи показывают: функция доходит до "Generating image 2/8" и получает `shutdown` — **Edge Function убивается по таймауту**. Одно изображение через OpenRouter занимает ~20-30 секунд. 8 изображений = ~3-4 минуты — это намного больше лимита Edge Function (~60 секунд).

### Как работает генерация карусели для Instagram (и почему она не ломается)

`ContentDetail.tsx` вызывает `generate-pipeline-images` **по одному слайду за раз** в цикле на фронтенде (параметр `slide_number`). Каждый вызов генерирует одно изображение и укладывается в таймаут.

### Проблема текущей реализации

Мы засунули цикл по 8 изображениям **в один вызов** `run-diagnostic-pipeline`. Это гарантированный таймаут.

### Решение: серверная цепочка вызовов (self-chaining)

Полностью серверное, пользователь может уходить со страницы.

#### 1. `run-diagnostic-pipeline/index.ts`
- Steps 1 + 1.5 остаются как есть
- Step 2: вместо цикла по всем изображениям — **fire-and-forget вызов** к отдельной функции `process-diagnostic-image` для первого плейсхолдера
- Немедленно возвращает ответ клиенту

#### 2. Новая функция `process-diagnostic-image/index.ts`
- Принимает `diagnostic_id` и `image_index`
- Генерирует ОДНО изображение (укладывается в таймаут)
- Загружает в Storage, обновляет `quiz_json` (заменяет конкретный плейсхолдер на URL)
- Обновляет `generation_progress.completed_images`
- Если остались ещё плейсхолдеры → **fire-and-forget вызов к самой себе** с `image_index + 1`
- Если все готовы → устанавливает `status: "ready"`
- Проверяет `checkCancelled` перед генерацией

#### 3. `DiagnosticDetail.tsx` — без изменений
Polling уже работает, отображает прогресс. Ничего менять не нужно.

#### 4. `config.toml`
Добавить секцию для новой функции.

### Итог

```text
run-diagnostic-pipeline          process-diagnostic-image
┌─────────────────────┐          ┌─────────────────────────┐
│ Step 1: Quiz        │          │ Generate image #0       │
│ Step 1.5: Card      │          │ Upload + update DB      │
│ Step 2: fire&forget ├─────────►│ fire&forget to self ────┤──►  image #1 ──► ... ──► image #7 ──► status: ready
│ Return immediately  │          └─────────────────────────┘
└─────────────────────┘
```

Каждый вызов обрабатывает ровно одно изображение → укладывается в таймаут. Весь процесс серверный.
