
## Проблема подтверждена

Сейчас это не баг генерации самих картинок и не баг HTML-шаблона:

- в очереди все 3 задачи генерации изображений завершились успешно;
- в `generated_html` письма все 3 маркера на месте: `{{image_placeholder_1}}`, `{{image_placeholder_2}}`, `{{image_placeholder_3}}`;
- но в `email_letters.image_placeholders` в базе сохранён только URL для `image_placeholder_3`, а у `1` и `2` он пустой.

Значит ломается именно этап **сохранения массива `image_placeholders`**, а не вставка в HTML.

## Корневая причина

В `src/pages/EmailBuilder.tsx` защита от перезаписи завязана на одном поле:

```ts
const [generatingPlaceholderId, setGeneratingPlaceholderId] = useState<string | null>(null);
```

Но пользователь может запустить **несколько генераций подряд**. Тогда происходит гонка:

1. стартуют 3 задачи;
2. первая завершается;
3. код делает `setGeneratingPlaceholderId(null)`, хотя 2 другие ещё идут;
4. autosave / manual save снова начинает писать `image_placeholders` в базу;
5. в базу улетает устаревший локальный массив, где есть не все картинки;
6. последняя завершившаяся задача “побеждает”, поэтому в итоге остаётся только один URL.

Дополнительно усиливает проблему то, что после завершения задачи фронтенд сейчас только делает `invalidateQueries`, но не фиксирует URL локально сразу же. Из-за этого окно для перезаписи становится ещё больше.

## Что нужно изменить

### 1) Сделать отслеживание не одной, а всех активных генераций изображений
Файл: `src/pages/EmailBuilder.tsx`

Заменить одиночный `generatingPlaceholderId` на структуру вида:

- `Set<string>` активных placeholder id
или
- `Record<string, boolean>`

Что это даст:
- можно одновременно корректно отслеживать 2–3 генерации;
- сохранение будет блокироваться, пока не завершится **последняя** задача;
- UI покажет загрузку именно на тех плейсхолдерах, которые реально генерируются.

### 2) Исправить guard в `save()`
Файл: `src/pages/EmailBuilder.tsx`

Сейчас логика опирается на `generatingPlaceholderRef.current`, который становится `false` слишком рано.

Нужно поменять правило на:
- **не сохранять `image_placeholders` вообще**, пока есть хотя бы одна активная генерация изображения.

Это уберёт перезапись базы устаревшим массивом во время фоновых задач.

### 3) После завершения каждой задачи сразу обновлять локальный state
Файл: `src/pages/EmailBuilder.tsx`

В `generatePlaceholderImage()` при поллинге задачи нужно запрашивать не только `status`, но и `result`, и при `completed`:

- взять `result.image_url`;
- сразу в `setImagePlaceholders(...)` проставить URL нужному `placeholderId`;
- только потом делать `invalidateQueries`.

Так UI и локальное состояние станут консистентными сразу, без ожидания повторной гидрации из базы.

### 4) Переделать передачу состояния загрузки в canvas
Файл: `src/components/email-builder/BlockCanvas.tsx`

Сейчас в canvas приходит один `generatingPlaceholderId`. Нужно передавать:

- либо `generatingPlaceholderIds: string[]`,
- либо функцию `isPlaceholderGenerating(id)`.

Тогда кнопки у нескольких плейсхолдеров смогут одновременно показывать корректный loading-state, и завершение одной задачи не “разблокирует” остальные раньше времени.

## Ожидаемый результат после правки

После запуска 3 генераций подряд:

- все 3 задачи могут спокойно завершиться;
- каждый завершившийся placeholder сразу получает свой URL в локальном состоянии;
- autosave не затрёт частично заполненный массив;
- после рефреша страницы в письме останутся все 3 изображения, а не только последнее.

## Технические детали

### Файлы
- `src/pages/EmailBuilder.tsx`
- `src/components/email-builder/BlockCanvas.tsx`

### Без изменений
- SQL/очередь для этой конкретной проблемы менять не нужно;
- HTML письма менять не нужно;
- edge function `generate-email-letter` уже возвращает успешный `image_url`, этого достаточно для фронтенда.

## Проверка после внедрения

1. Открыть текущее письмо.
2. Запустить генерацию сразу для 3 плейсхолдеров подряд.
3. Дождаться завершения всех 3 задач.
4. Убедиться, что:
   - все 3 изображения отобразились в canvas;
   - после ручного сохранения и авто-сохранения ничего не пропало;
   - после перезагрузки страницы все 3 URL остались в письме.
